# 04 · 前后端集成与接口契约

> 本文是前端与 voglander 后端之间的「接缝说明」。字段级明细见 [`../../api/voglander-api.md`](../../api/voglander-api.md)。

## 1. 契约总原则

前端必须**精确镜像后端**——不得自造字段或端点（见 `voglander/.cursor/rules/project-rule.mdc` 与 `vue-vben-admin/.cursorrules`）：

- web-antd 消费 voglander REST API；新增/变更字段须先在 cursor-rule 文件登记，**后端先行实现**，再做前端。
- 所有面向用户文本国际化（`module.entity.action`）。
- 时间：后端 VO 发毫秒时间戳，前端负责时区/格式化。

## 2. URL 重写与代理（关键，易踩坑）

前端 `requestClient` 的 `baseURL = apiURL = /api`（`.env.development` 的 `VITE_GLOB_API_URL=/api`）。各类 URL 的实际解析：

| API 函数里的 URL | 拼接 baseURL 后 | vite 代理 rewrite（剥离 `^/api`） | 后端实际收到（:8081） |
|------------------|-----------------|-----------------------------------|----------------------|
| `/api/v1/proxy/getPage` | `/api/api/v1/proxy/getPage` | → `/api/v1/proxy/getPage` | `/api/v1/proxy/getPage` ✅ |
| `/auth/login` | `/api/auth/login` | → `/auth/login` | `/auth/login` ✅ |
| `/user/info` | `/api/user/info` | → `/user/info` | `/user/info` ✅ |
| `/system/menu/all` | `/api/system/menu/all` | → `/system/menu/all` | `/system/menu/all` ✅ |
| `/zlm/api/media/list` | `/api/zlm/api/media/list` | → `/zlm/api/media/list` | `/zlm/api/media/list` ✅ |

vite 代理配置（`apps/web-antd/vite.config.mts`）：
```ts
server: { proxy: { '/api': {
  target: 'http://0.0.0.0:8081',
  changeOrigin: true,
  rewrite: (path) => path.replace(/^\/api/, ''),
  ws: true,
}}}
```
> 含义：业务里写 `/api/v1/...` 的端点，因为后端 `@RequestMapping` 本身就带 `/api/v1`，前端写全路径即可；`/auth`、`/user`、`/system` 这类后端**不带** `/api/v1` 前缀的端点，前端也写后端真实路径，代理只是统一剥离开发期的 `/api` 网关前缀。

## 3. 响应契约 · AjaxResult ↔ requestClient

后端统一返回：
```json
{ "code": 0, "msg": "操作成功", "data": <T> }
```
前端 `defaultResponseInterceptor` 配置：

| 配置 | 值 | 含义 |
|------|----|----|
| `codeField` | `'code'` | 状态码字段 |
| `successCode` | `0` | **0 = 成功**（非 200） |
| `dataField` | `'data'` | 取数据本体 |

`requestClient`（`responseReturn:'data'`）→ API 函数直接拿到 `data`，无需手动解包。失败码（如 500）进入错误/重认证拦截器。

| code | 语义 | 前端行为 |
|------|------|---------|
| 0 | 成功 | 返回 `data` |
| 401/token 失效 | 未认证 | `authenticateResponseInterceptor` 刷新或登出 |
| 300/500/其它 | 警告/错误 | `errorMessageResponseInterceptor` → `message.error(msg)` |

## 4. 鉴权流转

```
登录 POST /auth/login {username,password}
  → AjaxResult.data = { accessToken }
  → accessStore.setAccessToken(token)
后续请求：请求拦截器加 Authorization: Bearer <token>
  → 后端 JwtAuthInterceptor 校验，JwtUtils.getUserId(token)
token 过期：authenticateResponseInterceptor
  → doRefreshToken() POST /auth/refresh（baseRequestClient，带旧 token）
  → 写回新 token；刷新失败 → doReAuthenticate() 登出/弹窗
权限码：GET /auth/codes → string[] → hasAccessByCodes(['Module:Entity:Action'])
动态菜单：GET /menu/all（树形）、GET /menu/permissions（前端路由格式）
```

## 5. ZLM 多节点路由 · X-Node-Key

ZLM 是多节点部署，前端需指定目标节点：

1. 页面用 `NodeSelector` 选节点 → 同步 `nodeStore.setCurrentNodeKey()` **和** `utils/node-state.ts` 的 `setCurrentNodeKey()`
2. 请求拦截器检测 URL 含 `/zlm/api` → 从 `getCurrentNodeKey()` 读取 → 注入请求头 `X-Node-Key: <nodeKey>`
3. 后端据 `X-Node-Key` 把请求路由到对应 ZLM 服务器（zlm starter 的 `NodeSupplier`/`LoadBalancer`）
4. 未选节点时 `/zlm/api` 请求会 `console.error` 告警

> 为何双轨：请求拦截器是普通模块函数，无 Vue setup 上下文，不能直接用 Pinia store，故 `utils/node-state.ts` 持有一份模块级镜像。

## 6. 时间戳约定

- 后端 DO/DTO 用 `LocalDateTime`；**VO 出参统一 Unix 毫秒时间戳**，字段名以 `Time` 结尾（`createTime`/`updateTime`/`keepaliveTime`）。
- 前端拿到的是 `number`（毫秒），负责本地时区转换与展示格式化。
- 不要在前端假设后端发的是 ISO 字符串。

## 7. SSE 实时事件契约

| 维度 | 约定 |
|------|------|
| 端点 | `GET /api/v1/stream/events`（`text/event-stream`） |
| 鉴权 | `?token=<accessToken>`（URL query，EventSource 限制） |
| 订阅参数 | `?topics=device,session,clientcmd,alarm`（**前缀域**，缩短 URL） |
| 事件名 | SSE `event:` 名 = **完整 topic**（`device.register` / `clientcmd.ptz` / `session.invite` / `alarm.push` …）；前端逐个 `addEventListener(完整topic)` |
| topic 来源 | 完整 topic 列表由 `GET /api/v1/lab/client/config` 返回 |
| data | payload 本体的 JSON（**不含** `{topic,data}` 外层包裹）；含 `ts`（毫秒） |
| 去重 | 前端按 `(topic\|ts\|callId/deviceId/clientId/platformId/sn)`，规避单机 Redis 回环双投 |
| 重连 | `onerror` 3s 自动重连 |

## 8. 分页契约

- 列表分页统一 `POST .../getPage`，body 为查询条件 + `page`/`size`
- 返回 `*ListResp`：`{ total: number, items: T[] }`（后端 Manager `getPage` 默认 `createTime` 降序）
- 前端 VXE Grid `proxyConfig.ajax.query` 直接对接

## 10. 新增/变更接口的协作流程

1. 后端：在对应 Controller 按**模板方法规范**加端点（`*Req` 入参 → `WebAssembler` → DTO → Manager），统一 `AjaxResult` 返回。
2. 登记契约：在 cursor-rule 文件登记新增/变更字段。
3. 前端：在 `api/[module]/[entity].ts` 的 `namespace XxxApi` 内补类型与请求函数（`requestClient`），必要时更新桶导出 `index.ts`。
4. 页面：`data.ts` 补 Schema / 列，`list.vue`/`form.vue` 接入；补 i18n（`module.entity.action`）。
5. 校验：`pnpm check` + 类型检查通过。

> 字段级请求/响应示例，始终以 [`../../api/voglander-api.md`](../../api/voglander-api.md) 为单一事实来源。
